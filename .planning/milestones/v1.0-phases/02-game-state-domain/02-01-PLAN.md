    **MainDispatcherRule.kt** (package `com.mudita.sudoku.game`, under test sources):
    A JUnit 4 `TestWatcher` rule that replaces `Dispatchers.Main` with `UnconfinedTestDispatcher()` for ViewModel tests.

    ```kotlin
    package com.mudita.sudoku.game

    import kotlinx.coroutines.Dispatchers
    import kotlinx.coroutines.ExperimentalCoroutinesApi
    import kotlinx.coroutines.test.UnconfinedTestDispatcher
    import kotlinx.coroutines.test.resetMain
    import kotlinx.coroutines.test.setMain
    import org.junit.rules.TestWatcher
    import org.junit.runner.Description

    @OptIn(ExperimentalCoroutinesApi::class)
    class MainDispatcherRule(
        private val testDispatcher: kotlinx.coroutines.test.TestDispatcher = UnconfinedTestDispatcher()
    ) : TestWatcher() {
        override fun starting(description: Description) {
            Dispatchers.setMain(testDispatcher)
        }
        override fun finished(description: Description) {
            Dispatchers.resetMain()
        }
    }
    ```

    **FakeGenerator.kt** (package `com.mudita.sudoku.game`, under test sources):
    A class that mimics `SudokuGenerator.generatePuzzle(difficulty)` by returning a deterministic `SudokuPuzzle`. This avoids running real Sudoklify generation in ViewModel tests (which is slow and non-deterministic).

    The fake must:
    - Have a companion object `SOLUTION` — a valid complete 9x9 Sudoku solution stored as `IntArray(81)`. Use this known-valid solution:
      ```
      5,3,4,6,7,8,9,1,2,
      6,7,2,1,9,5,3,4,8,
      1,9,8,3,4,2,5,6,7,
      8,5,9,7,6,1,4,2,3,
      4,2,6,8,5,3,7,9,1,
      7,1,3,9,2,4,8,5,6,
      9,6,1,5,3,7,2,8,4,
      2,8,7,4,1,9,6,3,5,
      3,4,5,2,8,6,1,7,9
      ```
    - Have a companion object `BOARD` — the same solution with some cells zeroed out (e.g., zero out the first 20 non-given cells). The `givenMask` for these cells is false. Ensure at least 30 cells are given (easy-level) and at least 10 cells are empty for testing fill/pencil/undo operations.
    - `generatePuzzle(difficulty: Difficulty): SudokuPuzzle` returns `SudokuPuzzle(board = BOARD.copyOf(), solution = SOLUTION.copyOf(), difficulty = difficulty)` — always returns the same puzzle regardless of difficulty, just with the requested difficulty tag.
    - Provide a helper `fun emptyIndices(): List<Int>` that returns indices where `BOARD[i] == 0` — useful for tests to find fillable cells quickly.
    - Provide a helper `fun correctDigitAt(index: Int): Int` that returns `SOLUTION[index]` — useful for tests to know the right answer.
    - Provide a helper `fun wrongDigitAt(index: Int): Int` that returns a digit 1-9 that is NOT `SOLUTION[index]` — useful for error count tests.

    Do NOT extend or subclass `SudokuGenerator`. The `GameViewModel` constructor takes a `SudokuGenerator` with default construction — for testing, the ViewModel will need to accept the generator as a constructor parameter. The FakeGenerator approach: create a subclass of `SudokuGenerator` that overrides `generatePuzzle`. Since `SudokuGenerator` is NOT open, the FakeGenerator should instead be a standalone class. The `GameViewModel` will need to accept a lambda `suspend (Difficulty) -> SudokuPuzzle` or a simple interface. However, the research says `GameViewModel(generator: SudokuGenerator = SudokuGenerator())` — so the simplest approach is to make `FakeGenerator` extend `SudokuGenerator` by opening `generatePuzzle` in SudokuGenerator.

    **Alternative approach (preferred to avoid modifying Phase 1):** Create an interface or use a function type. But the research specifically shows `GameViewModel(generator: SudokuGenerator = SudokuGenerator())`. Since `SudokuGenerator.generatePuzzle` is a non-final method on a non-final class in Kotlin, it CAN be overridden without the `open` keyword only via Mockk. BUT Mockk in tests is fine. However, a cleaner approach: make `FakeGenerator` a proper subclass. Check if `SudokuGenerator` is `open` — it is NOT explicitly `open`, but in Kotlin all classes are `final` by default. The cleanest approach: use Mockk to mock `SudokuGenerator` in tests, OR make GameViewModel accept a functional parameter.

    **Decision:** Use a functional type in GameViewModel. Define `typealias PuzzleGenerator = suspend (Difficulty) -> SudokuPuzzle` and have GameViewModel accept this. Default implementation wraps `SudokuGenerator().generatePuzzle(difficulty)`. FakeGenerator is then a simple function.

    Actually, this is getting complex and will be resolved in Plan 02 when GameViewModel is created. For now, create FakeGenerator as a standalone utility class that provides the deterministic puzzle and helper methods. The exact ViewModel injection strategy will be decided in Plan 02.

    package com.mudita.sudoku.game

    import com.mudita.sudoku.puzzle.model.Difficulty
    import com.mudita.sudoku.puzzle.model.SudokuPuzzle

    class FakeGenerator {
        companion object {
            val SOLUTION = intArrayOf(/* 81 values */)
            val BOARD = intArrayOf(/* 81 values — some zeroed */)

            fun emptyIndices(): List<Int> = BOARD.indices.filter { BOARD[it] == 0 }
            fun correctDigitAt(index: Int): Int = SOLUTION[index]
            fun wrongDigitAt(index: Int): Int = (1..9).first { it != SOLUTION[index] }
        }

        fun generatePuzzle(difficulty: Difficulty): SudokuPuzzle =
            SudokuPuzzle(board = BOARD.copyOf(), solution = SOLUTION.copyOf(), difficulty = difficulty)
    }
