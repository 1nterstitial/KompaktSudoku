## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | White (#FFFFFF) | Cell backgrounds, screen background, inactive mode button fill, dialog background |
| Secondary (30%) | Black (#000000) | Grid lines, digit text, active mode button fill, error indicator border, all standard borders |
| Accent (10%) | Black (#000000) | Selected cell fill (solid black rect), active mode button background |
| Destructive | n/a | No destructive actions in Phase 4 |

**E-ink constraint:** The Mudita Kompakt display is monochromatic. The only colors in use are
`Color.Black` and `Color.White`. Do not introduce any grey values, tints, or color-tinted
schemes. MMD's `eInkColorScheme` enforces this — do not override it.

Accent reserved for:
- Selected cell background (solid black rect, GameGrid)
- Active input mode button background (Fill or Pencil, ControlsRow)

No other elements use the black-fill accent treatment.

**Source:** GameGrid.kt (Color.Black/White for selected cell, error indicator), ControlsRow.kt (Black/White mode toggle), CLAUDE.md §MMD Library (eInkColorScheme, no dynamicColorScheme)

## ResumeDialog — Detailed Contract

This is the sole new UI surface in Phase 4.

### Trigger condition
Displayed when `GameViewModel.showResumeDialog: StateFlow<Boolean>` emits `true`.
`showResumeDialog` is set to `true` in `GameViewModel.init` when `repository.loadGame()` returns
a non-null state.

### Dismissal
The dialog is NOT dismissible by tapping outside (no `onDismissRequest` that auto-dismisses to
a blank screen). Tapping outside the dialog, or pressing Back, is treated as "New Game" — the
same as pressing the "New Game" button. This prevents a stuck state where neither choice has
been made and the board is empty.

### Layout
```
+-----------------------------------+
|                                   |
|   Resume last game?               |  <- TextMMD, dialog title
|                                   |
|   [        Resume        ]        |  <- ButtonMMD, primary action
|   [       New Game       ]        |  <- ButtonMMD, secondary action
|                                   |
+-----------------------------------+
```

- Dialog container: `BasicAlertDialog` wrapping a `Column` with `Surface` background
- Container background: White (#FFFFFF)
- Container border: 1dp solid Black — must be visible on E-ink white background
- Container padding: 16dp vertical, 24dp horizontal (lg token from spacing scale)
- Button spacing between Resume and New Game: 16dp (md token)
- Each button: `ButtonMMD`, full width within dialog container, minimum 56dp height
- No divider lines between title and buttons

### Copy
| Element | Copy |
|---------|------|
| Dialog heading | "Resume last game?" |
| Primary action button | "Resume" |
| Secondary action button | "New Game" |

### Interaction
- "Resume" → calls `viewModel.resumeGame()` → dialog dismissed → board restored to saved state
- "New Game" → calls `viewModel.startNewGame()` (which clears saved state and calls `startGame(Difficulty.EASY)`) → dialog dismissed → new Easy puzzle begins
- Back press → treated as "New Game" (see Dismissal above)

### Ripple / Indication
Both ButtonMMD elements must have no ripple effect. `ButtonMMD` disables ripple by default via
`ThemeMMD`. Do not add any `indication` override that restores ripple.

### Implementation note
`BasicAlertDialog` is preferred over `AlertDialog` because `AlertDialog`'s built-in
`confirmButton`/`dismissButton` slots may add ripple to their internal button implementations.
`BasicAlertDialog` with `ButtonMMD` children is the safe, verified-ripple-free approach.

**Source:** 04-CONTEXT.md D-03/D-04, 04-RESEARCH.md Pattern 6, CLAUDE.md §MMD Library (ripple disabled)

## GameScreen Modification Contract

The `LaunchedEffect(Unit) { viewModel.startGame(Difficulty.EASY) }` in `GameScreen.kt` must
be guarded to prevent auto-starting a game when a resume prompt is pending.

### Before (Phase 3)
```kotlin
LaunchedEffect(Unit) {
    viewModel.startGame(Difficulty.EASY)
}
```

### After (Phase 4)
```kotlin
LaunchedEffect(Unit) {
    if (!viewModel.hasSavedGame()) {
        viewModel.startGame(Difficulty.EASY)
    }
}
```

Alternative: `GameViewModel.init` auto-starts a new game only when no saved state exists, making
the guard unnecessary in `GameScreen`. Either approach is acceptable — executor chooses based on
what keeps `GameScreen` thinner.

**Source:** 04-RESEARCH.md Pitfall 6 (double-init resume check)

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA (resume prompt) | "Resume" |
| Secondary action (resume prompt) | "New Game" |
| Dialog heading | "Resume last game?" |
| Loading state (no change from Phase 3) | "Generating puzzle\u2026" |
| Empty state | n/a — no empty state exists in Phase 4; a game is always either loading, in progress, or presenting the resume prompt |
| Error state — corrupt saved data | Silent recovery: treat as no saved game, start new Easy game with no dialog. No error copy shown to user. |
| Destructive confirmation | n/a — "New Game" from the resume dialog is not considered a destructive action requiring a second confirmation; the single-tap choice is sufficient |

**Copy rules:**
- Use sentence case, not title case ("New Game" is a proper noun phrase, "Resume" is a verb — both acceptable)
- No exclamation points — mindful design ethos, calm copy only
- Dialog heading ends with "?" — it is a question, not a command

**Source:** 04-CONTEXT.md D-03/D-04 (dialog copy confirmed in context decisions)

## Interaction States

| State | Visible Element | Copy / Treatment |
|-------|----------------|------------------|
| App launch — no saved game | None (proceeds directly to LoadingScreen then GameScreen) | "Generating puzzle\u2026" while loading |
| App launch — saved game exists | ResumeDialog modal over GameScreen | "Resume last game?" with Resume / New Game buttons |
| Player taps Resume | Dialog dismissed, board restored | Board reflects exact saved state; undo stack is empty |
| Player taps New Game | Dialog dismissed, new Easy game loads | "Generating puzzle\u2026" while new puzzle generates |
| App backgrounded mid-game | No visual change (save is silent) | Auto-save triggers in onStop(); player sees nothing |
| App backgrounded during loading (isLoading=true) | No visual change | Save is NOT triggered (guard condition); no corrupt state |
| App backgrounded on complete game (isComplete=true) | No visual change | Save is NOT triggered (guard condition); saved state is cleared |

---

## Anti-Patterns (do not implement)

- No spinner, progress bar, or animation for loading state — static TextMMD only (E-ink artifact prevention)
- No toast, snackbar, or banner for "game saved" — save is silent and automatic (D-01)
- No color-tinted dialog background — White only (monochromatic E-ink display)
- No ripple on dialog buttons — `ButtonMMD` or `indication = null` required
- No second confirmation before "New Game" — single tap is the confirmation (in-context dialog is sufficient)
- No "Cancel" or dismiss-only option on the resume dialog — player must choose Resume or New Game
